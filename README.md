## 실습 01 — 막연한 프롬프트 체험

**Block 1 · Day 1**

프롬프트: `"행사 신청 사이트 만들어줘"`

**결과**: 이름·이메일·연락처·참가 동기 폼. 신청하기 누르면 "완료" 메시지.

**개발자 3질문 (이 코드로는 못 답함)**
1. 신청자가 정원(30명)을 넘으면 어떻게 처리하나요?
2. 신청은 자동 승인인가요, 주최자가 검토하나요?
3. 취소는 언제까지 가능하고 대기자는 어떻게 되나요?

→ 기획서가 없으면 코드가 있어도 서비스를 만들 수 없다는 것을 체감하는 실습.

## 실습 09 — PRD 기반 MVP 구현·배포

**Block 9 · Day 2**

PRD 완성본 기반으로 Claude Code Desktop이 생성한 결과물.

| 파일 | URL | 설명 |
|------|-----|------|
| `index.html` | `/` | 참가자 신청 화면 |
| `admin.html` | `/admin` | 주최자 관리 화면 (승인·거절 HITL) |
| `vercel.json` | — | 클린 URL 라우팅 |

**실습 01과의 차이**
- 정원 표시 (남은 자리 실시간)
- 신청 후 "주최자 검토 중" 상태 명시 → HITL 인지
- 주최자가 직접 승인·거절 → HITL 체험
- LocalStorage로 두 화면 데이터 공유

**Vercel 배포 방법**
```bash
cd 실습09-mvp
# Vercel Dashboard에서 이 폴더를 루트로 지정
```

## 한계사항 (실습 09 기준)

이 프로토타입은 2일 과정 MVP이므로 아래 기능은 포함되지 않습니다.

| 한계 | 실제 서비스화 시 필요한 것 |
|------|------------------------|
| 인증 없음 (URL만으로 역할 분리) | 로그인·세션 관리 |
| 데이터 영구 저장 없음 (LocalStorage) | DB 연동 (Supabase 등) |
| 실시간 업데이트 없음 | WebSocket |
| 이메일 알림 없음 | 외부 이메일 API |

> **PM 관점**: 이 한계 목록을 아는 것이 역량입니다. "지금 안 되는 것"을 정의하고 다음 버전 로드맵을 세우는 것이 PM의 일입니다.
