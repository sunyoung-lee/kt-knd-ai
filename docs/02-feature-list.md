# 🧩 기능 리스트 (Feature List) — 최종

> 행사 신청 관리 플랫폼 — 주최자 중심 MVP
> 기반: docs/01-problem-definition.md · docs/02-feature-list-draft.md
> 우선순위: PM 확정 (MoSCoW)

## ✅ Must — 플로우가 막히면 안 되는 것 (MVP 필수)

| 기능 | 이유 |
|---|---|
| 행사 만들기 | 신청을 받으려면 행사가 먼저 존재해야 하는 플로우 출발점 |
| 신청 링크 생성 | 참가자를 신청 흐름으로 들여보내는 유일한 입구 |
| 신청 접수 | 신청을 받아 명단에 저장하는 핵심 — 이게 없으면 플랫폼이 성립 안 됨 |
| 내 신청 확인 페이지 | 참가자가 자기 신청을 다시 볼 창구가 없어 주최자에게 묻던 문제를 해소 |
| 셀프 수정/취소 | '수정/취소해주세요' 문의가 주최자에게 몰리는 핵심 통점을 직접 제거 |
| 명단 자동 반영 | 수동 처리 누락으로 생기던 '명단과 실제가 엇갈림'을 차단 |
| 신청 현황 대시보드 | 주최자가 문의 응대 없이 현재 신청 상태를 한눈에 파악 |
| CSV 내보내기 | 현장에서 쓸 최종 참가자 명단을 확보하는 출구 |

## 🔶 Should — 있으면 좋지만 MVP 없이도 동작

| 기능 | 이유 |
|---|---|
| 정원 초과 자동 차단 | 정원 관리 부담을 덜지만, 없어도 대시보드로 수동 마감 가능해 Must는 아님 |

## 🟦 Could — v2 이후

| 기능 | 이유 |
|---|---|
| 신청 항목 커스텀 | 행사별 맞춤 입력은 유용하나, MVP는 기본 항목(이름·연락처)으로 충분 |
| 대기자 자동 배정 | 취소 시 대기자 승계 자동화는 가치 크지만 동시성 처리 복잡도가 높아 후순위 |

---

## 범위 메모

- **MVP의 심장**: 내 신청 확인 페이지 · 셀프 수정/취소 · 명단 자동 반영 — 문제 정의서의 두 통점('문의 응대 폭주', '명단 엇갈림')을 동시에 해소.
- **Must 8개**가 주최자의 핵심 플로우(행사 생성 → 링크 공유 → 접수 → 셀프 변경/자동 반영 → 현황 확인 → 명단 내보내기)를 끊김 없이 완성.
- 정원/대기자 관련(Should·Could)은 동시성 이슈가 있어 MVP에서 분리, 단순 흐름을 먼저 검증.
