# 문제 정의서 (확정본)

## 사용자
- **정의**: 행사를 직접 여는 주최자와 신청하는 참가자를 잇는 양면(마켓플레이스) 사용자이며, 그중 운영 부담이 더 큰 주최자가 우선 대상이다.
- **근거**: 인터뷰에서 핵심 사용자를 "양쪽 모두(마켓플레이스)"로 꼽았고, 가장 큰 대가가 발생하는 쪽으로 "문의 응대 폭주(주최자)"를 지목했다.

## 문제
- **정의**: 신청 자체보다 '신청 후 확인/변경' 단계가 자동화되지 않아, 주최자에게 '문의 응대 폭주'가 발생하는 신청 과정의 번거로움이다.
- **근거**: "제 신청 됐나요? / 일정 바꿀 수 있나요?" 같은 반복 문의가 카톡·문자로 몰려 주최자가 일일이 응대해야 한다.

## 맥락
- **정의**: 주최자가 구글폼+시트 조합으로 행사를 운영하면서, 접수는 되지만 확인·변경 안내가 안 돼 모든 후속 소통을 카톡·문자로 처리하는 상황이다.
- **현재 대안과 한계**: 구글폼+시트는 접수만 될 뿐 발신(확인·리마인더)이 안 돼, 참가자가 자기 신청을 다시 볼 창구가 없고 후속 소통이 전부 주최자의 수동 카톡·문자로 역류한다.

## 기대 결과
- **정의**: 참가자가 스스로 자기 신청을 확인·변경·취소할 수 있게 되어, 주최자가 카톡·문자 응대 없이 행사 운영에만 집중할 수 있게 된다.
- **성공 판단 기준**: 주최자가 받는 행사별 문의 건수가 줄고, 참가자의 셀프 확인·변경 비율이 올라간다.
