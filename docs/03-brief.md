# 📝 기획서 (Product Brief) — 확정본

> 행사 신청 관리 플랫폼 — 주최자 중심 MVP
> 기반: docs/01-problem-definition.md · docs/02-feature-list.md
> 상태: PM 전 항목 동의 (Day 1 기획서 확정)
> 반영 범위: Must 기능 8개 (행사 만들기 · 신청 링크 생성 · 신청 접수 · 내 신청 확인 페이지 · 셀프 수정/취소 · 명단 자동 반영 · 신청 현황 대시보드 · CSV 내보내기)

## ① 문제 (Problem)

행사에 신청한 참가자가 신청 후 자기 신청을 스스로 수정/취소할 수 없어, '수정/취소해주세요' 문의가 주최자에게 몰리고, 주최자가 구글폼+시트로 수동 처리하는 과정에서 명단과 실제가 엇갈린다.

## ② 사용자 (User)

핵심 사용자는 행사를 여는 주최자(MVP 우선순위 1순위)이며, 구글폼+시트로 운영하며 신청 후 문의를 직접 응대한다. 참가자는 자기 신청을 스스로 확인·변경하려는 양면의 사용자로, 주최자의 부담을 줄이는 셀프서비스 대상이다.

## ③ 핵심 기능 (Key Features) — Must 8개

- 행사 만들기 — 신청을 받을 행사를 생성(플로우 출발점)
- 신청 링크 생성 — 참가자를 신청 흐름으로 들여보내는 입구
- 신청 접수 — 신청을 받아 명단에 저장
- 내 신청 확인 페이지 — 참가자가 자기 신청을 다시 볼 창구
- 셀프 수정/취소 — 참가자가 주최자 연락 없이 직접 변경
- 명단 자동 반영 — 수정/취소를 명단에 즉시 동기화
- 신청 현황 대시보드 — 주최자가 현황을 한눈에 파악
- CSV 내보내기 — 현장용 최종 명단 확보

## ④ KPI (숫자+기간)

- 핵심: 런칭 후 3개월 내, 행사당 수정/취소 관련 CS 문의 건수 50% 감소 (baseline은 런칭 시점 측정)
- 보조: 런칭 후 3개월 내, 참가자 셀프 수정/취소 처리율 70% 달성
- 보조: 런칭 후 3개월 내, 명단 정확도(현장 실제 인원 ↔ 명단 일치율) 95% 이상
※ 위 수치는 가설값(검증 필요), baseline 확보 후 확정.

## ⑤ 기대효과 (Expected Impact)

참가자가 '내 신청 확인 페이지'에서 스스로 수정/취소를 처리하고 그 결과가 주최자 명단에 자동 반영되어, '문의 응대 폭주'와 '명단 엇갈림'이 사라진다. 주최자는 신청이 늘어도 응대에 끌려가지 않고 운영에 집중할 수 있다.

## ⑥ HITL 포인트 (Human-in-the-Loop) & 운영 정책 결정

자동화에만 맡기지 않고 주최자(사람)가 개입·결정해야 하는 지점. 특히 (B)의 3가지는 답이 없으면 개발자가 구현을 시작할 수 없는 정책 결정이라, 기획서에서 방향을 확정하고 상세는 PRD(Day 2)에서 명세한다.

**(A) 사람이 개입하는 운영 흐름**
- 행사 만들기 시 행사 정보·신청 항목을 주최자가 직접 입력·검수
- 신청 링크를 어디에·언제 공유할지 주최자가 결정
- 셀프 수정/취소가 명단에 자동 반영된 뒤, 주최자가 대시보드에서 최종 현황을 확인
- CSV 내보내기 전 주최자가 명단을 점검하고 현장 사용 여부를 판단

**(B) 확정된 운영 정책 3종**
- ① 정원 초과 처리 → 정원 도달 시 자동 차단(추가 신청 막음). 자동 차단(Should) 미구현 시 대시보드에서 주최자가 수동 마감하는 폴백. 예외 처리 상세는 PRD.
- ② 자동 승인 vs 검토 → 자동 승인(신청 즉시 명단 확정, 주최자 승인 단계 없음). 수동 승인은 '응대 부담 감소'와 충돌하므로 MVP 제외, 검토 모드는 v2. 승인 플로우 명세는 PRD.
- ③ 취소 기한·대기자 → 취소는 행사 시작 전까지 셀프 허용, 대기자(취소 시 자동 승계)는 v2 제외. 취소 기한 컷오프·대기자 정책 상세는 PRD.

**(C) MVP 고려로 의도적으로 비운 것**
- 취소 후 정원 빈자리 자동 충원(=대기자 자동 배정)은 동시성 복잡도로 v2.
- 신청 항목 커스텀은 Could 기능이라 MVP 제외, 기본 항목(이름·연락처) 고정.
